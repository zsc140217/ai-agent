# 前端部署指南（微信云托管）

## 📋 部署前准备

### 1. 获取后端服务地址

1. 登录微信云托管控制台
2. 找到后端服务 `jblmj-ai-agent`
3. 复制访问地址，格式类似：
   ```
   https://jblmj-ai-agent-xxx.service.tcloudbase.com
   ```

### 2. 修改 nginx 配置

打开 `jblmj-ai-agent-frontend/nginx.conf` 文件，找到第 16 行：

```nginx
proxy_pass https://你的后端服务地址/api/;
```

**替换为实际地址**，例如：
```nginx
proxy_pass https://jblmj-ai-agent-xxx.service.tcloudbase.com/api/;
```

⚠️ **注意：**
- 地址末尾必须有 `/api/`
- 必须是 `https://` 开头
- 不要漏掉最后的斜杠 `/`

## 📦 打包前端项目

### 方法1：使用自动脚本（推荐）

1. 进入前端目录：
   ```
   cd jblmj-ai-agent-frontend
   ```

2. 双击运行 `打包前端.bat`

3. 生成 `frontend-deploy.zip`

### 方法2：手动打包

1. 进入 `jblmj-ai-agent-frontend` 文件夹

2. 选择以下文件（按住 Ctrl 多选）：
   ```
   ✅ src 文件夹
   ✅ public 文件夹
   ✅ index.html
   ✅ package.json
   ✅ package-lock.json
   ✅ vite.config.js
   ✅ Dockerfile
   ✅ nginx.conf
   ✅ .dockerignore
   ```

3. 右键 → 发送到 → 压缩(zipped)文件夹

4. 命名为 `frontend-deploy.zip`

---

## 🚀 部署到微信云托管

### 第1步：创建前端服务

1. 登录微信云托管：https://cloud.weixin.qq.com/

2. 进入你的环境（和后端同一个环境）

3. 点击 **"云托管"** → **"服务管理"**

4. 点击 **"新建服务"**

5. 填写服务信息：
   ```
   服务名称：jblmj-ai-agent-frontend
   部署方式：Dockerfile
   ```

6. 点击 **"下一步"**

### 第2步：上传代码

1. 代码来源：选择 **"本地代码上传"**

2. 点击 **"上传代码包"**，选择 `frontend-deploy.zip`

3. 等待上传完成

### 第3步：配置容器

```
📌 基础配置：
- Dockerfile 路径：Dockerfile
- 监听端口：80
- 流量策略：100%

📌 资源配置：
- CPU：0.25 核（前端不需要太多）
- 内存：512 MB
- 实例数量：1

📌 高级配置（可选）：
- 构建超时：1200 秒（20分钟）
- 启动超时：300 秒（5分钟）
```

### 第4步：开始部署

1. 点击 **"开始部署"**

2. 等待构建完成（约 5-8 分钟）

3. 构建过程：
   ```
   ⏳ 上传代码...
   ⏳ 构建镜像...
   ⏳ 安装 npm 依赖...
   ⏳ 执行 vite build...
   ⏳ 配置 nginx...
   ⏳ 启动容器...
   ✅ 部署成功！
   ```

## 🔍 常见问题

### Q1: 前端能访问，但 API 调用失败

**原因：** nginx.conf 中的后端地址配置错误

**解决方案：**
1. 检查 nginx.conf 中的 `proxy_pass` 地址
2. 确认后端服务地址正确
3. 重新打包并部署

### Q2: 页面空白或 404

**原因：** 构建失败或 nginx 配置问题

**解决方案：**
1. 查看构建日志，确认 `npm run build` 成功
2. 检查 Dockerfile 是否正确
3. 查看容器日志

### Q3: SSE 流式响应不工作

**原因：** nginx 缓冲配置问题

**解决方案：**
nginx.conf 中已经配置了：
```nginx
proxy_buffering off;
proxy_cache off;
chunked_transfer_encoding off;
```
如果还有问题，检查后端服务是否正常

### Q4: 跨域问题

**原因：** 前后端域名不同

**解决方案：**
使用 nginx 反向代理（已配置），前端通过 `/api/` 访问后端，不会有跨域问题
